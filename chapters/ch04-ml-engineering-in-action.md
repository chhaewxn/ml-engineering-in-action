# **우리가 공부한 것 🙇**

## CHAPTER 4. 의사소통과 프로젝트 규칙 논의

> 한 문장: ML 프로젝트의 성패는 모델 성능이 아니라 의사소통에서 갈린다. 비즈니스와의 소통, 실험의 경계 설정, 비즈니스 규칙 통합, 그리고 기대치 관리까지 -- 코드 밖의 일이 코드 안의 일보다 중요하다!

---

### 4.1 의사소통: 문제 정의

- **프로젝트 의사소통의 순서**
    - 문제 이해 → 연구 후 단계 논의(업데이트 회의) → 실험 후 단계(주제 전문가/UAT 리뷰) → 개발 스프린트 리뷰(기술 전문가가 아닌 대상을 위한 진행 보고서) → MVP 리뷰(UAT를 사용한 전체 데모) → 사전 프로덕션 리뷰(UAT를 사용한 최종 데모)

---

**사업부와 소통할 때 염두에 둘 것**

1. 소소한 오류는 무시하고 **필수 세부 사항**에 중점 두기
2. 솔루션 구축 방법이 아닌 **산출물**에 중점 두기
3. 가장 좋은 방법은 프론트엔드 개발자처럼 최종 상태를 시뮬레이션하는 **와이어프레임 활용**하기

---

**브레인스토밍 중에 염두에 둘 것**

1. 데이터 과학 팀은 브레인스토밍에서 나온 모든 아이디어를 환영하되, 추가 비용을 고려해야 함
2. 아이디어를 무시해서는 안 되지만, 모든 아이디어를 실험 계획에 포함하는 것도 허용해서는 안 됨
3. 엔지니어링은 회의에서 배제할 것 -- '수요 예측에서 LSTM을 쓸지, LGBM을 쓸지'와 같은 논의는 AI 팀 외의 사람들을 소외시키게 됨
4. 이 단계의 모든 논의는 **코드가 한 줄도 작성되기 전에** 이뤄져야 함

<aside>
💡 "대부분의 ML 실무자는 ML이 무엇인지 이해하지 못하는 사람들의 아이디어를 바탕으로 무엇이 가능하고 불가능한지 즉석에서 결정해버리는 경향이 있습니다." -- 필자의 조언: 그냥 경청하라.
</aside>

---

**프로젝트 회의와 납기일**

- 프로젝트 회의는 프로젝트 마일스톤에 맞춰 진행해야 함 → 정기적인 요일과 시간을 정하는 주간 미팅을 지양할 것
- 팀원들이 내린 결정은 설사 실험 단계에서 내린 결정이더라도 '신성한 것으로 간주해야 함'
    - "외부의 목소리와 도우려는 사람들을 참여시키면, 팀원들이 함께 구축한 포용적인 소통 환경이 약화됩니다"
- 예상 납기일을 정할 때 여유분을 넣지 마라 → 정직한 납기일을 제시하고, 상황에 따라 조정이 필요할 수 있음을 설명할 것. 여유분을 포함해서 예상 납기일을 제시하면, 점차적으로 사업부의 신뢰를 떨어트림

---

**연구 후 단계 논의에서 염두에 둘 것**

1. 탐색한 선택지를 전부 언급하지 말 것 -- 결과가 훌륭하더라도 구축하는 데 2년이 걸릴 아이디어는 말도 꺼내지 말 것
2. 여러 옵션 중, 더 마음에 드는 것은 **회사에서** 결정하게 하라. ML 엔지니어가 직접 결정하지 않고 아래의 것들만 제시할 것:
    - 각 옵션으로 무엇을 할 수 있는지
    - 각 옵션을 통해 가능한 것과 불가능한 것이 각각 무엇인지
    - 각 옵션에 대해 언제쯤 프로토타입을 볼 수 있을지

<aside>
💡 "실험 단계에서 정말 근사하던 것이 유지 관리할 때는 악몽이 된다는 점을 꼭 기억하세요."
</aside>

---

**실험 후 단계(주제 전문가/UAT 리뷰)에서 염두에 둘 것**

- 실험 단계를 마무리하는 PT의 목표: 핵심 피처의 목업을 보여주는 것
- 전체 팀 상황 보고 회의에서는 **현재 상태 데모**를 보여주기 위해 노력해야 함 → 엔지니어가 간과했을 세부 사항에 대해 피드백을 받을 수 있음

---

**MVP 리뷰에서 염두에 둘 것**

- **'코드 완료'의 정의**
    1. 코드가 테스트되고, 단위 테스트와 통합 테스트를 통과
    2. 시스템이 프로덕션 규모 데이터를 사용하는 평가 환경에서 원활하게 작동
    3. 합의하에 계획한 피처가 모두 완료되고 설계된 대로 작동
- ML 구현에서 metric은 훌륭한 도구이지만, 솔루션이 문제를 질적으로 해결하는지 판단하는 가장 좋은 척도는 **내부 사용자와 전문가의 폭넓은 지식을 활용하는 것**
- 릴리즈는 금요일에 하지 말 것
- 배포한 모델의 효과를 검증할 때에는 인내심을 가질 것 -- 솔루션 성능에 대한 데이터가 충분히 수집된 이후에나 통계적으로 유효한 판단을 내릴 수 있음

---

### 4.2 시간 낭비하지 않기: 크로스펑셔널 팀과의 회의

**프로젝트 단계별 회의의 핵심 질문은 각각 다르며, 벗어난 논의는 시간 낭비**

| 회의 단계 | 핵심 질문 |
| --- | --- |
| 실험적 업데이트 회의 | 이 문제를 우리가 해결할 수 있는가? (모델 세부 사항, 추가 Algorithm 등의 논의는 불필요) |
| SME 프로토타입 리뷰 | 리소스를 투입할 가치가 있는가? (실제 데이터로 시연하여 SME의 주관적 평가를 받아야 함) |
| 개발 진행 리뷰 | 이전에 지적된 문제가 해결되었는가? (말로 주장하지 말고, 동일한 데이터로 직접 증명) |
| MVP 리뷰 | 요청한 대로 빌드했는가? (프로젝트 외부 인원이 UAT를 수행해야 함) |
| 사전 프로덕션 리뷰 | 원래 계획과 최종 결과물이 일치하는가? (모든 피처를 QA 환경에서 실시간 검증) |

---

**"이 문제를 해결할 수 있는가?"는 기술적 가능성이 아니라 주어진 제약 내에서 가능한가를 묻는 것**

- 실패 인정은 빠를수록 좋음. 투입한 시간과 에너지가 클수록 포기가 어려워짐 (매몰 비용)
- 조기 중단하지 못하면 최악의 경우 회사 전반의 ML 불신으로 이어짐

---

**MVP 구축 후 바로 프로덕션에 출시하면 반드시 후회함**

- MVP 합의 이후 반드시 **코드 강화 스프린트**를 거쳐야 함: 테스트, 모니터링, 로깅, 프로덕션용 코드 정비

---

**UAT(User Acceptance Test) 평가 원칙**

- UAT 평가는 프로젝트에 관여하지 않은 사람이 수행해야 하며, 점수는 반드시 normalization
- 프로젝트에 감정이 이입된 SME의 평가는 Confirmation Bias가 발생
- UAT에서 문제가 단 하나도 나오지 않았다면, 평가자가 제대로 평가하지 않았을 가능성이 높음
- 리뷰 편향 유형: 극단값만 주는 사람, 평균으로 일관하는 사람, 과도하게 좋은 점수를 주는 사람
- 각 평가자의 점수 범위 내에서 normalization한 후 분석해야 신뢰할 수 있는 결과를 얻음

---

**개발 중 예상치 못한 변경이 발생하면**

- ML 프로젝트에서 변경은 일상적인 일이며, Agile을 적용하면 영향을 최소화할 수 있음
- API 계약에 영향을 미치면 프런트엔드 팀과 상의, 비즈니스 로직에 영향을 미치면 SME에게 전달
- **닌자처럼 은밀히 해결하는 것은 금지.** 아무에게도 알리지 않고 조용히 수정하면 나중에 예상치 못한 문제가 발생할 확률이 매우 높음

<aside>
💡 변경이 생기면 숨기지 말고 공유하라. 은밀한 수정은 단기적으로 편하지만 장기적으로 재앙이 된다.
</aside>

### 4.3 실험 한계 설정

**실험 단계의 경계는 어떤 모델을 사용할지 결정하는 것**

- 시간 제한을 설정하여 프로토타입에 과도한 시간 투입을 방지해야 함
- 실험 단계에서 필터링 로직, 사용자 지정 가중치, 추가 데이터셋 생성 등 확장 피처까지 구현하면 핵심 목표(모델 선택)를 해결하지 못한 채 시간만 소모됨
- 계획 수립 단계에서 나온 아이디어 대부분은 실험에서 구현되지 않는다는 점을 팀 전체가 인식해야 함

---

**실험 코드는 끔찍해도 OK → 최종 Solution에서 확장은 NO**

- 실험 코드는 스크립트 수준으로 빠르게 작성: 차트, print문, 하드코딩된 값 등 잘못된 코딩 관행이 허용됨
- 클래스, 인터페이스, 디자인 패턴 등 정교한 코드 구조는 이 단계에서 시간 낭비
- 실험 코드의 역할은 팀이 프로덕션용 코드를 작성할 때 **참고**하는 것이지, 그 위에 덧붙이는 것이 아님

---

**프로토타입의 예측 품질만으로 접근법을 선택하면 안 됨**

- 결과가 가장 좋은 것을 선택하는 것은 가장 흔하면서도 위험한 판단
- 예측 품질 외에 반드시 검토해야 할 항목: 구축 기간, 코드 복잡도, 재훈련 비용, 팀이 해당 Algorithm/언어/플랫폼을 유지보수할 역량이 있는지, 문제 발생 시 수정 속도, 확장 가능성, 테스트 가능성, 감사 가능성
- **예측력이 아닌 프로덕션 구현 가능성**을 기준으로 평가해야 성공적인 Solution과 허울뿐인 Solution을 구별할 수 있음

---

**네 가지 개발 방식**

| 방식 | 설명 | 권장 여부 |
| --- | --- | --- |
| **TDD + FDD 혼합** | 모듈식 코드 + 테스트 가능성 + 피처 중심 점진적 개발. 단순한 프로젝트는 TDD 위주, 복잡한 프로젝트는 FDD 원칙으로 Feature Sprawl을 억제하면서 TDD의 테스트 가능성 활용 | 유일하게 유효 |
| **PDD (기도 주도 개발)** | 연구 없이 블로그나 타인의 코드를 복사하여 해결하려는 방식. 테스트 중인 모델이 단 1개이고 팀에 해당 문제 경험자가 없다면 PDD를 하고 있는 것 | 피해야 함 |
| **CDD (혼돈 주도 개발)** | 실험/프로토타이핑을 건너뛰고 바로 구현. 스파게티 코드와 전면 재작업의 원인 | 피해야 함 |
| **RDD (이력서 주도 개발)** | 문제 해결이 아닌 참신한 기술 사용 자체가 목적. 모델 코드 대비 Feature Engineering 코드 비율이 4배 미만이면 RDD를 의심할 것 | 피해야 함 |

<aside>
💡 실험 코드의 역할은 "참고"이지 "확장"이 아니다. 프로토타입에서 근사했던 것이 프로덕션에서 악몽이 되는 이유는 대부분 이 경계를 무시했기 때문이다.
</aside>

### 4.4 비즈니스 규칙 혼돈에 대한 계획 수립

**규칙의 혼돈**

- 사용자가 불쾌하지 않도록 보호하거나 부적절한 콘텐츠를 필터링하는 것은 올바르게 구현하기가 가장 어려움
- 출시 전 제한사항을 발견하지 못하면 회사의 브랜드가 훼손될 수 있음
- 추천 목록에 불쾌감을 줄 만한 상품이 포함될 수 있음을 인지해야 함

---

**대비하는 유일한 방법: '지렛대와 손잡이' 방식으로 코드를 설계하는 것**

- 이를 '예상'하고, 전달된 구성(passed-in configuration)을 통해 적용하거나 수정할 수 있는 함수나 메서드로 설계
- 코드를 고치는 방식이 아닌 **외부에서 모델의 행동을 제어할 수 있도록** 설계 → 코드를 전면 재작성하거나 조정할 필요가 없어짐
- 솔루션 개발에 많은 시간을 할애하기 전에 '잠재적인 제한 사항'을 이해하는 것이 가장 좋음
- 제한 사항을 미리 파악하면 전체 아키텍처와 피처 엔지니어링에 영향을 미치며, 조정 시간을 단축하고 사후 수정의 비용을 줄일 수 있음

---

**비즈니스 규칙 vs 나머지 규칙**

| 구분 | 성격 | 처리 방법 |
| --- | --- | --- |
| **비즈니스 규칙** | 법적/정책적으로 반드시 지켜야 하는 '기본 사양'. 모델 설계 단계부터 무조건 포함되어야 함 | 가중치, 조건부 논리, 불리언 스위치 등 어떤 방식이든 결과물에 반드시 반영 |
| **나머지 규칙** | 있으면 좋지만 당장 완벽할 필요는 없는 실험적 기능 | A/B 테스트할 수 있는 구성을 구축하거나, MVP에 넣기엔 중요도가 낮은 기능은 향후 작업으로 간주 |

---

**주제 전문가에게 성능 수정 권한을 부여하기**

- 비즈니스 상황이 바뀌어 개발자가 다시 '하드코딩'을 하는 것을 방지해야 함
- 시스템을 중단하지 않고도 시스템 동작을 빠르게 변경하는 방식으로 접근
- 제어 기능도 추가하여 적절한 검증 절차를 거쳐 수정하도록 해야 함

---

**백업 계획과 콜드 스타트 문제**

- 데이터 품질에 문제가 생길 것을 예상하여 구축해야 함. 사용자 개인화 데이터셋을 사용할 수 없는 경우 '백업'할 수 있는 안전한 일반 데이터셋 확보가 중요
- **콜드 스타트 문제**: 피처 벡터가 비어있거나 과거 데이터가 없는 등 데이터가 충분하지 않은 경우. 이를 인식하고 대체 방안을 미리 마련하는 것이 중요

<aside>
💡 비즈니스 규칙은 코드에 하드코딩하지 말고, 외부 설정으로 제어할 수 있게 설계하라. "지렛대와 손잡이" 방식이 유지보수의 핵심이다.
</aside>

---

### 4.5 결과에 대해 말하기

**마법의 답변: '알고리즘은 가르쳐준 것만 안다'**

- 모델이 이상한 결과를 내면 일반인은 다양한 질문과 화를 냄
- 알고리즘의 기능과 성능을 설명하지 말고, '알고리즘은 가르쳐준 것만 안다'라는 답변이 효과적
- 이는 '알고리즘이 틀린 게 아니라, 우리가 준 데이터에 상관관계가 잘못되었거나 데이터가 부족했다'는 의미
- ML 측면과 관련된 방식으로 인과 개념을 명확히 설명해 문제와 그 근본 원인을 알려줘야 함

---

**ML이 할 수 없는 것을 설명하자**

- 데이터 수집에는 한계가 존재함 (주관적 선호도 같은 개인적 특성, 처리 비용, 복잡성 등)
- ML 모델은 보이지 않는 변수까지 고려하지 못함
- 현실 세계에서는 변수가 무한히 많으나, 데이터에서는 그렇지 못하기에 확실하게 예측할 수는 없음

---

**기대치를 명료하고 분명하게 관리하자**

- 훈련할 정보가 없으면 예측도 할 수 없다는 한계를 프로젝트 초기에 미리 설명해야 함
- 비현실적인 기대를 깨뜨리고, 나중에 실망하고 좌절할 일을 없앰
- 기대치를 미리 설명하면 프로젝트 팀장이 창의적인 방식으로 해결하거나, 빠르게 포기하도록 길을 정하게 됨
- **'약속은 지키고 초과 달성하는 것이 늘 최선'** -- 보수적으로 말한 후 실제 결과를 초과로 달성하는 것이 성공한 프로젝트로 평가를 받음

---

**다섯 살짜리 아이에게 하듯 설명하자**

- 의사소통의 목표는 '이해받는 것'
- 기술에 대한 '우월한 어조'는 소통을 단절시키고 팀원의 조롱과 분노를 일으킴
- 전문 용어를 사용해 소외되는 대화를 하지 말고, 그들이 잘 아는 용어를 사용하여 설명하고, 복잡한 주제는 우화와 예시를 곁들여 설명해야 함

<aside>
💡 ML 프로젝트에서 가장 중요한 기술은 코딩이 아니라 커뮤니케이션이다. "알고리즘은 가르쳐준 것만 안다"는 한 마디가 10분짜리 기술 설명보다 효과적이다.
</aside>

---

### 4.6 요약

1. 비기술적인 언어와 결과 솔루션에 기반하여 소통하면 ML 프로젝트 목표 달성에 가까워짐
2. 마일스톤을 설정하여 중간 과정을 시연하면 재작업과 예기치 않은 기능 결함은 줄어듦
3. 애자일 개발 방식처럼 목표를 짧고 엄격히 관리하여 연구, 실험, 프로토타이핑의 시간을 단축시킴
4. 비즈니스 규칙과 기대치를 미리 정의하고 통합하여 조정과 설계를 해야 함

**Q1. 실험 코드의 품질과 속도 사이에서 실무적으로 어떻게 균형을 잡는가?**

A. 실험 단계에서는 속도가 우선이므로 코드 품질을 의도적으로 낮추는 것이 올바른 선택이다. 핵심은 실험 코드를 프로덕션 코드의 기반으로 삼지 않는 것이다. 실험 코드는 "이 접근법이 작동하는가?"에 대한 답만 빠르게 얻기 위한 도구이며, 프로덕션 코드는 실험 결과를 참고하여 처음부터 새로 작성해야 한다.

## 단어집 📓

| **용어** | **정의** |
| --- | --- |
| UAT(User Acceptance Test) | 최종 사용자 또는 주제 전문가가 시스템이 요구 사항을 충족하는지 확인하는 테스트 |
| SME(Subject Matter Expert) | 특정 도메인에 대한 깊은 지식을 가진 주제 전문가 |
| 와이어프레임(Wireframe) | 최종 상태를 시뮬레이션하기 위해 만드는 UI/UX 설계 초안 |
| TDD(Test-Driven Development) | 테스트를 먼저 작성하고, 그 테스트를 통과하는 코드를 작성하는 개발 방식 |
| FDD(Feature-Driven Development) | 피처(기능) 단위로 점진적으로 개발하는 방식 |
| PDD(Prayer-Driven Development) | 연구 없이 블로그나 타인의 코드를 복사하여 해결하려는 방식 (지양해야 함) |
| CDD(Chaos-Driven Development) | 실험/프로토타이핑을 건너뛰고 바로 구현하는 방식 (지양해야 함) |
| RDD(Resume-Driven Development) | 문제 해결이 아닌 참신한 기술 사용 자체가 목적인 개발 방식 (지양해야 함) |
| 콜드 스타트(Cold Start) | 피처 벡터가 비어있거나 과거 데이터가 없는 등 데이터가 충분하지 않은 상태 |
| 일일 스탠드업 미팅(Daily Scrum) | 매일 짧게(10분 이내) 업무 현황을 브리핑하고 이슈를 공유하는 미팅 |
| 몽키 패치(Monkey Patch) | 프로그램 실행 중(Runtime)에 코드의 동작을 동적으로 수정하거나 확장하는 기술 |
| Shelfware(셸프웨어) | 구매했으나 설치하지 않거나 사용하지 않고 방치된 소프트웨어 |
| Confirmation Bias(확증 편향) | 자신의 기존 믿음이나 기대에 부합하는 정보를 선호하고, 반대되는 정보를 무시하는 인지 편향 |
| Feature Sprawl(피처 확산) | 핵심 목표를 벗어나 불필요한 기능이 계속 추가되는 현상 |
| 매몰 비용(Sunk Cost) | 이미 투입되어 회수할 수 없는 비용으로, 의사결정 시 무시해야 하지만 실제로는 포기를 어렵게 만드는 요인 |
